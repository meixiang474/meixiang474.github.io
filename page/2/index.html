如何理解MVVM

组件化基础

很久以前就有组件化
  • asp, jsp, php已经有组件化了
  • nodejs中也有类似的组件化
数据驱动视图
  • 传统组件, 只是静态渲染, 更新还要依赖于操作DOM
  • 数据驱动视图———–vue MVVM, react setState
vue MVVM

view(dom) —— vue, viewModel(dom listener, directives) ——- model

Vue响应式

  • Object.defineProperty实现响应式

  • 监听对象, 监听数组

  • 复杂对象, 深度监听

  • 深度监听需要递归到底, 一次性计算量大

  • 无法监听新增属性和删除属性, (Vue.set, Vue.delete)

  • 无法原生监听数组, 需要做特殊处理

虚拟DOM和diff

  • vdom是实现vue和react的重要基石
  • diff算法是vdom中最核心, 最关键的部分
  • DOM操作非常耗费性能
  • 虚拟dom过程
    • 创建vnode
    • patch(container, vnode)将虚拟dom渲染
    • var newVnode 创建新的虚拟节点
    • patch(vnode, newVnode) 更新
    • patch(newVnode, null) 销毁
  • 用js模拟DOM结构
  • 新旧vnode对比, 得到最小更新范围, 最后更新DOM
  • 数据驱动视图的模式下, 有效控制DOM操作
  • vdom的存在价值: 数据驱动视图, 控制DOM操作

diff算法

  • 树diff的事件复杂度o(n^3)
    • 第一, 遍历tree1, 第二, 遍历tree2
    • 第三排序
    • 算法不可用
  • 优化事件复杂度到o(n)
    • 只比较同一层级, 不再跨级比较
    • tag不相同, 则直接删掉重建, 不再深度比较
    • tag和key都相同, 则认为是相同节点, 不再深度比较

snabbdom源码解读

  • h函数返回一个vnode
  • patch函数
    • 第一个参数不是vnode, 则创建一个空的vnode, 关联到这个dom元素
    • 两个参数是相同的vnode(tag和key都相同)执行patchVnode
      • vnode对比—-patchVnode函数
        • 设置element, 给新的vnode复制成旧的vnode的element
        • 如果两者children相等, 则return
        • 如果旧有children, 新无children, 移除旧的children, 移除旧的text
        • 如果新旧都有children执行 updateChildren
        • 新有children, 旧无children, 将旧的text设置为空, 再将新的children添加进去
        • 如果新旧的text不相等, 先移除旧的children, 设置新的text
    • 不相同的vnode, 直接删掉重建
  • updateChildren
    • 开始和开始
    • 结束和结束
    • 开始和结束
    • 结束和开始
    • 以上均调用patch函数
    • 未命中
      • 新的key能否对应上oldCh中的某个key
        • 没对应上, 直接重建插入
        • 对应上了, 拿到key对应的节点, 判断标签是否一样
          • 不一样则重建插入
          • 一样的话, 则patch

模板编译

  • 模板编译为render函数, 执行render函数返回vnode
  • 基于vnode再执行patch 和diff
  • 使用webpack vue-loader, 会在开发环境下编译模板

组件渲染和更新过程

  • 初次渲染过程
    • 解析为render函数(或在开发环境已经完成, vue-loader)
    • 触发响应式, 监听data属性getter setter
    • 执行render函数, 生成vnode, patch(elem, vnode)
  • 更新过程
    • 修改data, 触发setter
    • 重新执行render函数, 生成newVnode
    • patch(vnode, newVnode)
  • 异步渲染
    • 汇总data修改, 一次性更新视图
    • 减少DOM操作次数, 提高性能

前端路由原理

  • hash的特点
    • hash变化会触发网页跳转, 即浏览器的前进和后退
    • hash变化不会刷新页面, spa必须的特点
    • hash永远不会提交到server端
    • window.onhashchange
  • history模式
    • 用url规范的路由, 但跳转不刷新页面
    • history.pushState
    • window.onpopstate
    • 需要后端支持

性能优化原则

  • 多实用内存, 缓存或其他方法

    • 使用更快的网络, CDN
  • 渲染更快
    • CSS放在head, JS放在body最下面
    • 尽早开始执行JS, 用DOMContentLoaded触发
    • 懒加载
    • 对DOM查询进行缓存
    • 频繁操作DOM, 合并到一起插入DOM结构
    • 节流throttle, 防抖debounce

js基础-渲染机制

Posted on 2020-02-25

从输入url到渲染出页面的整个过程

  • 加载资源
  • 渲染页面

window.onload和DOMContentLoaded的区别

  • 前者资源全部加载完才执行, 包括图片
  • 后者是在dom渲染完成就执行, 图片可能尚未下载

知识点

  • 加载资源的形式
    • html代码
    • 媒体文件
    • js css
  • 加载资源的过程
    • DNS解析: 域名 —> IP地址
    • 浏览器根据IP地址向服务器发起http请求
    • 服务器处理请求, 并返回给浏览器
  • 渲染页面的过程
    • 根据HTML代码生成DOM树
    • 根据CSS代码生成CSSOM
    • 将DOM树和CSSOM整合成Render Tree
    • 根据Render Tree渲染页面
    • 遇到script则暂停渲染, 优先加载并执行JS代码, 完成再继续
    • 直至把Render Tree渲染完成
